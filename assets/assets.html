공개 자산을 취급

이 절은 JavaScript, CSS 및 이미지 등의 응용 프로그램의 정적 자원의 제공을 포함합니다.

Play 2.0에서 public 자원을 제공하는 것은 다른 HTTP 요청에 서비스를 제공하는 것과 거의 동일합니다. 일반적인 자원과 동일한 라우팅을 사용합니다 : 클라이언트에 CSS, JavaScript 및 이미지 파일을 배포 할 수있는 컨트롤러 / 액션 경로를 사용합니다.

public / 폴더

규칙에 따라 public 자산은 응용 프로그램의 public 폴더에 저장되어 있습니다. 이 폴더는 다음과 같이 구성되어 있습니다 :

public 
 └ 자바 스크립트 나
  └ StyleSheets
  └ images

이 폴더 구성을 따른다면 시작하는 것은 간단하지만 작동 방식을 이해하고 있으면, 변경해도 괜찮습니다.

어떻게 public 에셋은 패키지됩니까?

빌드 프로세스 중에 public 폴더의 내용이 처리되고 어플리케이션의 클래스 패스에 추가합니다. 응용 프로그램을 패키징 할 때 ( public / 경로 아래에있는)이 파일은 응용 프로그램의 JAR 파일로 패키지됩니다.

자산 컨트롤러

Play 2.0은 공개 에셋을 제공하는 임베디드 컨트롤러가 포함되어 있습니다. 기본적으로이 컨트롤러는 캐시 기능, ETag, gzip 압축, JavaScript mi​​nify 지원이 제공됩니다.

컨트롤러는 controllers.Assets 등의 기본 Play JAR에서 사용할 수 있으며, 2 개의 파라미터를 가지는 at 작업을 정의합니다.

Assets . at ( folder :  String , file :  String )

folder 매개 변수는 고정되어 있으며, 작업을 통해 관리되는 디렉토리를 정의해야합니다. file 매개 변수는 일반적으로 동적으로 요청 경로에서 추출됩니다.

conf / routes 에서 Assets 컨트롤러의 전형적인 설정은 아래에서 보여드립니다 :

GET   / assets / * file Assets.at ( "public", file)

Note that we define the * File dynamic part that Will Match the * regular expression. So for example, if you send this request to the server :

GET / assets / javascripts / jquery . js

라우터는 다음 매개 변수를 사용하여 Assets.at 작업을 시작합니다 :

controllers . Assets . at ( "public" ,  "javascripts / jquery.js" )

이 작업은 파일을 찾아 있으면 제공합니다.

외부 "public"리소스 설정을 정의하려면, sbt에 가르쳐야한다는 점을주의하십시오.

GET   / assets / * file Assets.at ( "public", file)
GET / liabilities / * file Assets.at ( "foo", file)

Build.scala에 다음을 추가하십시오.

playAssetsDirectories <+ = baseDirectory /  "foo"

공개 자원의 리버스 라우팅

routes 파일에 매핑 된 모든 컨트롤러와 마찬가지로 리버스 컨트롤러가 controllers.routes.Assets 에 만들어집니다. 공개 자원을 얻기 위해 필요한 URL을 리버스 할 때 사용합니다. 템플릿의 예는 다음과 같습니다 :

<script  src = "@ routes.Assets.at (" javascripts / jquery . js ")" > </ script>

다음 결과를 생성합니다.

<script  src = "/ assets / javascripts / jquery.js" > </ script>

리버스 루트 때 folder 매개 변수를 지정하지 않을 것에 유의하십시오. 이것은 루트 파일 folder 매개 변수가 고정 된 Assets.at 작업에 대해 하나의 매핑을 정의하고 있기 때문입니다. 따라서 명시 적으로 지정할 필요는 없습니다.

그러나 Assets.at 작업에 2 개의 매핑을 정의하는 경우 다음과 같이하십시오 :

GET   / javascripts / * file Assets.at ( "public / javascripts", file)
GET / images / * file Assets.at ( "public / images", file)

리버스 라우터를 사용하려면 두 매개 변수를 지정해야합니다.

<script  src = "@ routes.Assets.at (" public / javascripts "," jquery . js ")" > </ script> 
<image  src = "@ routes.Assets.at (" public / images "," logo . png ")" >

Etag 지원

Assets 컨트롤러는 자동으로 ETag HTTP 헤더를 관리합니다. ETag 값은 자원 이름과 파일의 최근 수정에서 생성됩니다. (리소스 파일이 파일에 포함되어 있다면, JAR 파일의 마지막 수정 날짜가 사용됩니다.)

Web 브라우저가이　ETag 를 지정하여 요청하면 서버는 304 NotModified 에 응답 할 수 있습니다.

Gzip 지원

같은 이름을 가진 자원 . gz 확장자를 사용하는 것이 발견되었을 경우, Assets 의 컨트롤러는 적절한 HTTP 헤더를 추가하여 제공합니다.

Content - Encoding : gzip

Cache-Control 명령 추가

일반적으로 ETag를 사용하면 적절한 캐시를 가질 수 있습니다. 특정 자원에 대한 사용자 지정 Cache-Control 헤더를 지정하려면, 당신의 application.conf 파일에 지정할 수 있습니다. 예를 들어 :

# Assets configuration 
# ~ ~ ~ ~ ~ 
"assets.cache. / public / stylesheets / bootstrap.min.css" = "max-age = 3600"

관리 자산

play는 기본적으로 app / assets 폴더에 저장된 모든 에셋을 컴파일합니다. 이 컴파일 프로세스는 모든 관리 자산을 변경에 상관없이 클린하고 다시 컴파일합니다. 프런트 엔드 기술에 거는 의존성 추적은 매우 귀찮아서, 이것이 가장 안전한 전략입니다.

대량의 관리 자산을 취급하는 경우,이 전략은 매우 느리게 수 있음에주의하십시오. 이러한 이유로, 변경된 파일 및 의존 할 것으로 예상되는 파일 만 다시 컴파일하는 방법이 있습니다. 다음을 설정에 추가하여,이 실험적인 기능을 활성화 할 수 있습니다 : incrementalAssetsCompilation : = true

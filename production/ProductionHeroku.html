Heroku에 배치

Heroku 는 Web 어플리케이션의 구축 및 배치를 효율화하는 클라우드 애플리케이션 플랫폼입니다.

Procfile 만들기

Heroku를 이용함에있어 응용 프로그램의 루트 디렉터리에 Procfile 라는 특별한 파일을 작성해야합니다. 다음과 같은 내용의 텍스트 파일을 만듭니다.

web : target / start - Dhttp . port = $ { PORT } $ { JAVA_OPTS }

응용 프로그램을 git에 저장

개발중인 응용 프로그램의 git 저장소를 생성하십시오.

$ git init
$ git add . 
$ git commit - m "init"

Heroku에서 새 응용 프로그램을 만들

사전에 Heroku 계정 등록, heroku gem 설치가 필요합니다.

$ heroku create - Stack cedar
 Creating Warm - frost - 1289. ..  done , Stack is cedar
http : / / warm-1289.herokuapp.com / | git@heroku.com : warm-1289.git 
자식 remote Heroku Added

응용 프로그램이 전개 된 것을 확인

는 배포 된 응용 프로그램의 상태를 확인하자.

$ Heroku PS
 Process        State                Command 
------------   ------------------   -------------- -------- 
web . 1          그대로 for  10s           target / start

Web 프로세스를 시작합니다. 다음은 자세한 내용을 확인하기 위해 로그를 살펴 보도록하자.

$ heroku logs
 2011 - 08 - 18T00 : 13 : 41 + 00 : 00 heroku [ web . 1 :  Starting Process with Command `target / start` 
2011 - 08 - 18T00 : 14 : 18 + 00 : 00 app [ web . 1 :  Starting on Port : 28328 
2011 - 08 - 18T00 : 14 : 18 + 00 : 00 app [ web . 1 :  Started . 
2011 - 08 - 18T00 : 14 : 19 + 00 : 00 heroku [ web . 1 :  State Changed from Starting to 그대로
 ...

문제없는 것 같습니다. 또한 heroku open 명령을 실행하면 브라우저가 시작하고 전개 한 Web 어플리케이션에 액세스 할 수 있습니다.

만약 Web 프로세스를 시작하지 않고, Error H14 (No web processes running) 라는 오류가 기록되는 것 같으면, web dynamo를 추가해야 할 것이다.

heroku ps : scale web = 1 
heroku restart

조정

간단한 응용 프로그램도 용량이 Heroku의 slug size 최대 인 100MB를 빨리 넘어 버립니다.
이유는. ivy 캐시가 slug에 포함되어 버리고 있기 때문입니다. 사용자 지정 heroku "build pack"을 이용하면이 문제는 쉽게 해결할 수 있습니다. build pack는 원격 git 저장소에 저장되는 여러 스크립트를 세트로 한 것입니다. 스크립트는 git에서 받고, 컴파일시에 실행됩니다. 표준 build pack은 여기 에서 공개되어 있습니다.

구체적으로는 GitHub에서 build pack 저장소를 fork하여 bin / compile 스크립트 후반을 다음과 같이 변경합니다.

... 
mkdir - p $ cache_dir
 for DIR in $ CACHED_DIRS ;  do 
  RM - rf $ CACHE_DIR / $ DIR
  mkdir - p $ CACHE_DIR / $ DIR
  cp - r $ DIR /.   $ CACHE_DIR / $ DIR
echo "-----> Dropping ivy cache from the slug : $ DIR" 
rm - rf $ SBT_USER_HOME /. ivy2
 done

같은 방법으로 다른 문제 조치 Heroku에서 Scala 템플릿을 컴파일 할 때 문자 코드로 UTF8을 명시해야한다 -에도 대응할 수 있습니다. 이것도 build pack을 다음과 같이 수정합니다.

... 
echo "-----> Running : sbt $ SBT_TASKS" 
test - e "$ SBT_BINDIR" / sbt . boot . properties && PROPS_OPTION = "-Dsbt.boot.properties = $ SBT_BINDIR / sbt.boot.properties" 
HOME = "$ SBT_USER_HOME_ABSOLUTE" java - Xmx512M  - Dfile . encoding = UTF8 - Duser . home = "$ SBT_USER_HOME_ABSOLUTE"  - Divy . default . ivy . user . dir = "$ SBT_USER_HOME_ABSOLUTE/.ivy2" $ PROPS_OPTION - jar "$ SBT_BINDIR" / $ SBT_JAR $ SBT_TASKS 2 > & 1  | sed - u 's / ^ / /' 
if  [  "$ {PIPESTATUS [*]}"  ! =  "0"  ];  Then 
  echo "! Failed to build app with SBT $ SBT_VERSION " 
  exit  1 
fi 
...

실제로 작동하는 build pack은 여기 에 있습니다.

마지막 단계 : build pack 주소를 Heroku의 config var로 설정합니다.

heroku config : add BUILDPACK_URL = 'git@github.com : joergviola / heroku-buildpack-scala.git'

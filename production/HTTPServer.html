프론트 엔드가되는 HTTP 서버 설치

HTTP 포트를 80로 설정하면 응용 프로그램을 쉽게 독립형 서버로 배포 할 수 있습니다.

$ start - Dhttp . port = 80

프로세스를 80 번 포트에 바인딩하기 위해 root 권한이 필요할 수 있습니다.

그러나 동일한 서버에서 여러 응용 프로그램을 호스팅하거나 확장 성과 복원력의 관점에서 응용 프로그램의 여러 인스턴스를 시작하고, 그들을로드 밸런싱 할 수있을 것이다. 이 경우 프런트 엔드 HTTP 서버를 사용할 수 있습니다.

lighttpd를 사용

다음은 lighttpd 를 프런트 엔드 Web 서버로 구성하는 예입니다. 동일한 용도에 Apache를 사용할 수도 있지만, 필요한 것이 가상 호스팅 및로드 균형 조정 기능 만한다면 lighttpd 쪽이 설정이 간단해서 추천합니다.

server . modules =  ( 
      "mod_access" , 
      "mod_proxy" , 
      "mod_accesslog"  
) 
... 
$ HTTP [ "host" ]  = ~  "www.myapp.com"  { 
    proxy . balance =  "round-robin" proxy . server =  (  " / "  => 
        (  (  "host"  =>  "127.0.0.1" ,  "port"  =>  9000  )  )  ) 
}
 
$ HTTP [ "host" ]  = ~  "www.loadbalancedapp.com"  { 
    proxy . balance =  "round-robin" proxy . server =  (  "/"  =>  (  
          (  "host"  =>  "127.0.0.1" ,  " port "  =>  9001  ),  
          (  "host"  =>  "127.0.0.1" ,  "port"  =>  9002  )  )  
    ) 
}

Apache를 사용

다음은 Apache httpd server 를 표준 설정 Play 응용 프로그램의 전단에 배치하는 간단한 설정 예입니다.

LoadModule proxy_module modules / mod_proxy . so
 ... 
< VirtualHost  * : 80 > 
  ProxyPreserveHost  On 
  ServerName www . loadbalancedapp . COM
   ProxyPass   / excluded ! 
  ProxyPass  / http : / / 127.0.0.1:9000 / 
  ProxyPassReverse  / http : / / 127.0.0.1:9000 / 
</ VirtualHost >

리버스 프록시에 대한 자세한 설정

프런트 엔드 HTTP 서버를 이용하는 경우, Play 응용 프로그램에서하면 HTTP 요청이 HTTP 서버에서 전송되는 것처럼 보입니다. 평소와 같은 컴퓨터에서 Play 응용 프로그램 및 리버스 프록시를 시작하는 경우, Play에서 요청이 127.0.0.1에서 전송되는 것처럼 보입니다.

리버스 프록시가있는 HTTP 서버는 프록시 대상 Play 응용 프로그램에 요청을 보내는을 알리기 위해 특별한 헤더를 부여 할 수 있습니다. 대부분의 Web 서버는 X-Forwarded-For라는 헤더의 첫 번째 인수에 요청하는 클라이언트의 IP 주소를 붙여 보냅니다. 그런 Web 서버를 이용하고있는 경우, Play 응용 프로그램 XForwardedSupport을 설정함으로써 Play가 request.remoteAddress를 리버스 프록시의 IP 주소에서 요청자가 클라이언의 IP 주소로 변경하여줍니다. 그러나이 기능을 이용하기 위해서는 응용 프로그램 설정에서 리버스 프록시의 IP 주소를 모두 집어 넣을 필요가 있습니다.

이 설정을 변경 한 경우에도 host 헤더는 변경되지 않고, 리버스 프록시가 보낸 그대로의 내용이 있습니다. 그래서 예를 들어 Apache 2.x를 사용하는 경우 다음 설정을 할 좋습니다.

ProxyPreserveHost on

이 설정을하면 Play 응용 프로그램에 전송되는 Host 헤더가 클라이언트에서 보낸 원래의 Host 헤더의 내용과 동일합니다. 이처럼 XForwardedSupport과 ProxyPreserveHost 두 설정을 결합, 응용 프로그램이 클라이언트에 직접 공개되는 것을 振る舞える 수 있습니다.

만약, 리버스 프록시에 일부 요청 만 Play 응용 프로그램에 처리하도록하려면 일부 경로를 프록시 제외 설정을 추가하십시오.

ProxyPass  / excluded !

Aapche를 역방향 프록시로 사용하여 다운 타임없이 Play 응용 프로그램 업데이트

기본적인 아이디어는 Play 응용 프로그램의 인스턴스를 두 시작하고 프런트 엔드 프록시 서버에 그들을로드 균형시킵니다. 한 인스턴스가 떨어져있는 경우에는 실행중인 다른 인스턴스에만 요청을 전송하도록합니다.

먼저 같은 Play 응용 프로그램을 두 번 실행합니다. 하나는 포트 9999 번, 다른 하나는 포트 9998에서 시작합니다.

$ start - Dhttp . port = 9998 
$ start - Dhttp . port = 9999

다음 Apache의로드 밸런싱을 설정합니다.

Apache 설정 파일에 다음과 같은 설정을 설명합니다.

<VirtualHost  mysuperwebapp . com : 80 >
  ServerName mysuperwebapp.com
  <Location  / balancer-manager >
    SetHandler balancer-manager
    Order Deny, Allow
    Deny from all
    Allow from. mysuperwebapp.com
  </ Location> 
  <Proxy  balancer : / / mycluster >
    BalancerMember http://localhost:9999
    BalancerMember http://localhost:9998 status = + H
  </ Proxy> 
  <Proxy * >
    Order Allow, Deny
    Allow From All
  </ Proxy>
  ProxyPreserveHost On
  ProxyPass / balancer-manager!
  ProxyPass / balancer :/ / mycluster /
  ProxyPassReverse / http://localhost:9999/
  ProxyPassReverse / http://localhost:9998/
</ VirtualHost>

특히 중요한 것은, balancer :/ / mycluster 라는 부분입니다. 이것이 부하 분산의 정의입니다. + H 옵션은 두 번째 Play 응용 프로그램이 대기 계임을 선언합니다. 그러나 동시에이 대기 시스템을로드 밸런스 대상으로 추가 할 수 있습니다.

Apache는 선언 된 클러스터의 상태를 표시하는 기능도 제공합니다. 현재 상태를 표시하기 위해 브라우저에서 / balancer-manager에 액세스합시다.

Play 완전히 무국적 아키텍처되어 있기 때문에 두 클러스터 간의 세션 공유에 고민 할 필요가 없습니다. 그 덕분에 특히 어려움없이 3 개 이상의 인스턴스에 확장시킬 수 있습니다.

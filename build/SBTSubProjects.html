하위 프로젝트

복잡한 프로젝트를 하나의 Play 응용 프로그램에 정리할 필요가 없습니다. 큰 프로젝트는 작은 응용 프로그램으로 나눌 수 있습니다. 또한 응용 프로그램에서 독립적 인 로직은 Java 나 Scala 라이브러리에 자르는 것도 좋을 것입니다.

자세한 내용은 "멀티 프로젝트 빌드에 대한 SBT 문서" (원문) 을 참조하십시오. 또한 서브 프로젝트를 정의함에있어 가장 기본적인 것으로서 하위 프로젝트의 빌드 파일이라는 것은 없습니다. 부모 빌드 파일에 하위 프로젝트를 정의합니다.

라이브러리를 하위 프로젝트로 자르는

응용 프로그램에서 간단한 "라이브러리 프로젝트"를 자르는 것이 있습니다. project / Build.scala 다음과 같은 sbt 프로젝트 정의를 추가하십시오.

import SBT . _
 import  Keys . _
 import  PlayProject . _

object  ApplicationBuild  extends  Build  {

  val appName          =  "my-first-application" 
  val appVersion       =  "1.0"

  val appDependencies =  Seq ()
  
  val mySubProject =  Project ( "my-library" , file ( "modules / myLibrary" ))

  val main =  PlayProject ( 
    appName , appVersion , appDependencies
   ). dependsOn ( mySubProject ) 
}

이제 modules / myLibrary 디렉토리에 하위 프로젝트를 정의했습니다. 하위 프로젝트는 보통 sbt 프로젝트의 일종이며, 표준 디렉토리 구성을 따릅니다.

modules
  └ myLibrary
     └ src
        └ Main
           └ Java
           └ 스칼라

빌드 설정에서 하위 프로젝트를 사용하는 경우, 각각의 프로젝트를 개별적으로 컴파일, 테스트, 실행 할 수 있습니다. Play 콘솔 projects 명령을 실행하면 모든 프로젝트가 표시됩니다.

[ my - first - application ] $ projects
 [ info ]  In File : / Volumes / Data / gbo / myFirstApp / 
[ info ]  	 *  my - first - application
 [ info ]  	   my - library

현재 프로젝트를 전환하려면 project 명령을 사용하십시오.

[ my - first - application ] $ Project my - library
 [ info ]  Set Current Project to my - library
 >

Play 응용 프로그램을 개발 모드에서 실행중인 경우 종속 서브 프로젝트도 자동으로 다시 컴파일됩니다. 하위 프로젝트의 컴파일 오류가 브라우저에서 확인할 수 있습니다.

Web 어플리케이션을 여러 프로젝트에 분할

Play 응용 프로그램은 기본 설정에 따른 일반 sbt 프로젝트이기도하므로, 서로 의존성을 갖게 할 수 있습니다.

설정 방법은 하위 프로젝트의 경우와 매우 비슷합니다. 하위 프로젝트를 PlayProject 로 정의하면 OK입니다.

import SBT . _
 import  Keys . _
 import  PlayProject . _

object  ApplicationBuild  extends  Build  {

  val appName =  "zenexity.com" 
  val appVersion =  "1.2"

  val common =  PlayProject ( 
    appName +  "-common" , appVersion , path = file ( "modules / common" ) 
  )
  
  val website =  PlayProject ( 
    appName +  "-website" , appVersion , path = file ( "modules / website" ) 
  ). dependsOn ( common )
  
  val adminArea =  PlayProject ( 
    appName +  "-admin" , appVersion , path = file ( "modules / admin" ) 
  ). dependsOn ( common )
  
  val main =  PlayProject ( 
    appName , appVersion
   ). dependsOn ( 
    website , adminArea
   ) 
}

이제 Web 사이트 관리자 사이트로 구성된 main 프로젝트를 만들 수있었습니다. 또한 두 프로젝트가 common 모듈에 의존하고 있습니다.

main 프로젝트를 다시 컴파일하고 테스트 할 때 사용하는 프로젝트가 다시 컴파일되어 테스트되도록하려면, "aggregate"절을 추가해야합니다.

  val main =  PlayProject ( 
    appName , appVersion
   ). dependsOn ( 
    website , adminArea
   ) aggregate ( 
    website , adminArea
   ) 
}

Play 2.0 응용 프로그램의 구조

응용 프로그램의 표준 파일 구성

Play 응용 프로그램의 파일 구성은 최대한 단순 중시하고 표준화되어 있습니다. Play 응용 프로그램의 표준 파일 구성은 다음과 같이되어 있습니다.

app                       →  응용 프로그램의 소스 코드 
 └ Assets                 →  다른 언어로 컴파일 된 자산의 소스 파일 
    └ StyleSheets         →  일반적으로 LESS CSS 소스 
    └ 자바 스크립트 나         →  보통  CoffeeScript  소스 
 └ controllers            →  응용 프로그램의 컨트롤러 
 └ models                 →  응용 프로그램의 비즈니스 논리 층 
 └ views                  →  템플릿 
conf                      →  설정 파일 
 └ Application . conf       →  기본 설정 파일 
 └ Routes                 →  루트 정의 
public                    →  공개 자산 
 └ StyleSheets            → CSS 파일 
 └ 자바 스크립트 나            →  JavaScript  파일 
 └ images                 →  이미지 파일 
Project                   → SBT 설정 파일 
 └ build . properties       → SBT 프로젝트의 기본 파일 
 └  Build . 스칼라            →  응용 프로그램의 빌드 스크립트 
 └ plugins . SBT            → SBT 플러그인 
lib                       →  관리되지 않는 라이브러리 의존성 
logs                      →  기본 로그 위치 
 └ Application . log        →  기본 로그 파일 
target                    →  Play  의해 생성 된 파일 
 └ 스칼라 - 2.9 . 1               
    └ cache              
     └ classes             →  컴파일 된 클래스 파일 
    └ classes_managed     →  Play  관리 클래스 파일  (템플릿 등) 
    └ resource_managed    →  Play  관리 리소스  ( less 등) 
    └ src_managed         →  생성 중간 코드  (템플릿 등) 
test                      →  단위 테스트와 기능 테스트 소스 폴더

app / 디렉토리

app 디렉토리에 실행 가능한 모든 코드가 포함됩니다. Java 나 Scala 코드 나 템플릿, LESS와 CoffeeScript 같은 다른 언어로 컴파일 된 자산의 소스 파일 등 모든입니다.

app 디렉토리에 3 개의 패키지가 있습니다. 각각 MVC 패턴에 등장하는 세 가지 구성 요소에 해당합니다.

  • app / controllers
  • app / models
  • app / views

물론 app / utils 같은 독자적인 패키지 만들기도 문제 없습니다.

Play 2.0 controllers , models , views 는 패키지 이름은 느슨한 약관에 필요하다면 변경할 수 있습니다 (예를 들어, 모든 패키지에 com.yourcompany 접두사를 켜거나).

이 세 가지 외에도 필요는 없지만 app / assets 라는 디렉토리가 있습니다. 여기에는 LESS 와 CoffeeScript 같은 다른 언어로 컴파일 된 자산의 소스 파일을 놓습니다.

public / 디렉토리

public / 디렉토리에 저장된 자원은 이른바 정적 콘텐츠입니다, Web 브라우저에 그대로 전송됩니다.

이 디렉토리에는 기본적으로 3 개의 하위 디렉토리가 있습니다. 각 이미지 파일, CSS 파일, JavaScript 파일을 켭니다. 모든 Play 응용 프로그램이 유사한 구성되도록 정적 콘텐츠는이 디렉토리에 넣어두면 좋을 것입니다.

참고 응용 프로그램을 새로 만든 상태에서 public / 디렉토리 / assets 라는 URL 경로에 매핑되어 있습니다. 이것은 언제라도 변경할 수 있습니다. 또한 정적 콘텐츠를 여러 디렉토리에 분산시킬 수 있습니다.

conf / 디렉토리

conf 디렉토리에는 응용 프로그램의 설정 파일을 선택합니다. 주요 설정 파일은 다음 두 가지입니다.

  • application.conf 는 응용 프로그램의 기본 설정 파일. 표준 설정 항목을 모두 포함하고 있습니다.
  • routes 라는 루트 정의 파일.

만약 응용 프로그램 고유의 설정을 추가하려면 application.conf 으로 설정 항목을 추가하는 것이 좋습니다.

또한 라이브러리 자체 설정 파일은 conf 디렉토리에 배치하는 것이 좋습니다.

lib / 디렉토리

lib 디렉토리는 필요 없지만, 빌드 시스템의 단독에두고 싶은 jar 파일 등 모든 관리되지 않는 라이브러리 의존성을 포함합니다. jar 파일이 디렉토리에 배치하는 것만으로 응용 프로그램의 클래스 경로에 추가됩니다.

project / 디렉토리

project 디렉토리는 sbt 빌드 정의를 포함합니다.

  • plugins.sbt 이 프로젝트에서 사용 sbt 플러그인 정의 파일입니다.
  • Build.scala 이 응용 프로그램의 빌드 스크립트입니다.

target / 디렉토리

target / 디렉토리 Play 빌드 시스템에서 생성 된 모든 파일이 들어 있습니다. 무엇이 생성되는지 기억해두면 좋을 것입니다.

  • classes / 는 Java 나 Scala 코드에서 컴파일 된 클래스 파일이 들어 있습니다.
  • classes_managed / 는 Play가 관리하는 클래스 (예를 들어, 라우터 및 템플릿 시스템에 의해 생성 된 클래스)이 들어 있습니다. IDE를 사용하는 경우이 디렉토리에 클래스 패스를 통해두면 편리 하겠지요.
  • resource_managed / 은 Play 의해 생성 된 리소스가 포함되어 있습니다. 예를 들어, LESS CSS와 CoffeeScript에서 트랜스 컴파일 된 CSS 파일과 JavaScript 파일이 디렉토리에 배치됩니다.
  • src_managed / 는 템플릿 시스템이 생성 한 Scala 코드 등 Play 의해 생성 된 소스가 포함되어 있습니다.

잘 사용한다. gitignore 파일

Play 의해 생성 된 디렉토리는 버전 관리 시스템에 무시하도록 할 것입니다. Play 응용 프로그램에서 전형적인 . gitignore 파일의 내용을 다음과 같습니다.

logs
project / project
project / target
target
tmp

데이터베이스 에볼루션

관계형 데이터베이스를 사용하는 경우, 스키마 진화를 추적 · 정리하는 방법이 필요합니다. 특히 다음과 같은 경우에는 데이터베이스 스키마 변경을 추적하는 고급 도구를 사용 할 경우가 있습니다.

  • 팀에서 개발을 진행하고 있으며, 각 멤버가 스키마에 대한 모든 변경을 알고 있어야하는
  • 응용 프로그램을 프로덕션 서버에 배포하는 경우에는 데이터베이스 스키마를 안전하게 업그레이드 할
  • 여러 대의 컴퓨터에서 개발을 진행하고 있으며, 모든 시스템에서 데이터베이스 스키마를 동기화 할

에볼루션 스크립트

Play는 에볼루션 스크립트가 데이터베이스 진화를 추적합니다. 이 스크립트는 그냥 SQL로 작성된 응용 프로그램의 conf / evolutions / {데이터베이스 명} 라는 디렉토리에 저장되어 있습니다. 기본 데이터베이스에 진화를 적용 할 경우, 스크립트의 위치는 conf / evolutions / default 입니다.

첫 번째 스크립트는 1.sql 두 번째 스크립트는 2.sql , ...와 같이 이름을 지정합니다.

각 스크립트는 각각 두 개의 파트로 구성되어 있습니다.

  • Ups 파트는 필요한 스키마 변환 방법 설명
  • Downs 부분은 위의 변환을 취소하는 방법 설명

운동에 기본 응용 프로그램을 시작하는 첫 번째 진화 스크립트를 작성하여 보자.

# Users 스키마
 
# ---! Ups
 
CREATE TABLE User  ( 
    id bigint ( 20 ) NOT NULL AUTO_INCREMENT , 
    email varchar ( 255 ) NOT NULL , 
    password varchar ( 255 ) NOT NULL , 
    fullname varchar ( 255 ) NOT NULL , 
    isAdmin boolean NOT NULL , 
    PRIMARY KEY ( id ) 
);
 
# ---! Downs
 
DROP TABLE User ;

보시다시피 Ups 파트와 Downs 부분은 코멘트로 구분해야합니다.

데이터베이스가 application.conf 로 설정되어 있고, 에볼루션 스크립트가 있으면, 진화가 자동으로 활성화됩니다. 진화를 비활성화하려면 evolutionplugin = disabled 라는 설정을하십시오. 예를 들어, 자기 부담으로 데이터베이스를 설치하는 같은 자동화 테스트를 작성 같은 경우는 테스트 환경에서 진화를 비활성화하는 설정을 실시합시다.

진화가 활성화되어 있으면, Play는 개발 모드 요청마다 실전 모드 응용 프로그램을 시작하기 전에 스키마의 상태를 확인합니다. 개발 모드에서는 데이터베이스 스키마가 최신이 아닌 경우 적절한 SQL 스크립트를 실행하여 데이터베이스 스키마를 동기화하라는 오류 페이지가 표시됩니다.

SQL 스크립트의 내용에 문제가 없으면이 Apply evolutions 버튼을 클릭하면 즉시 스크립트를 실행 할 수 있습니다.

동시에 행해진 변경을 동기화

만일 프로젝트에 두 명의 개발자가 있다고합니다. 개발자 A는 새로운 테이블을 필요로하는 새로운 기능을 담당하고 있습니다. 그래서 개발자 A는 다음과 같은 2.sql 을 만듭니다.

# Post 추가
 
# ---! Ups 
CREATE TABLE Post  ( 
    id bigint ( 20 ) NOT NULL AUTO_INCREMENT , 
    title varchar ( 255 ) NOT NULL , 
    content text NOT NULL , 
    postedAt date NOT NULL , 
    author_id bigint ( 20 ) NOT NULL , 
    FOREIGN KEY ( author_id ) REFERENCES User ( id ), 
    PRIMARY KEY ( id ) 
);
 
# ---! Downs 
DROP TABLE Post ;

Play이 에볼루션 스크립트 개발자 A의 데이터베이스에서 실행합니다.

한편, 개발자 B는 User 테이블의 스키마 변경을 요하는 새로운 기능을 담당하고 있습니다. 따라서 개발자 B는 다음과 같은 2.sql 을 만듭니다.

# User 변경
 
# ---! Ups 
ALTER TABLE User ADD age INT ;
 
# ---! Downs 
ALTER TABLE User DROP age ;

개발자 B는 담당하고있는 기능을 구현 끝난 후, 그것을 커밋합니다 (그들이 Git를 사용하고 있다는 가정). 그러면 개발자 A는 계속함에있어서 동료들이 수행 한 작업을 병합해야합니다. 그는 git pull을 수행하지만, 병합에 충돌이 있습니다.

Auto - merging db / evolutions / 2.sql 
CONFLICT ( add / add ) :  Merge conflict in DB / evolutions / 2.sql 
Automatic MERGE failed ; fix Conflicts and  Then COMMIT the result .

두 사람의 개발자가 각각 2.sql 을 만들었 기 때문에, 개발자 A는이 파일의 내용을 병합해야합니다.

이 에볼루션 스크립트가 나타내는 데이터베이스 버전 2는 먼저 개발자 A가 적용 개정 2과는 내용이 다릅니다.

이 때, Play는이 두 개정 내용의 차이를 감지하고, 우선 현재 적용되어있는 이전 버전의 수정 2로 변경을 취소, 그리고 새로운 방식의 개정 2 스크립트를 적용합니다.

불일치 상태

가끔 에볼루션 스크립트에 오류가있어 적용에 실패하는 경우도있을 것이다. 그 경우, Play는 데이터베이스 스키마를 불일치 상태로 표시합니다. 그리고 계속하기 전에 문제를 수동으로 해결하도록 당신에게 촉구합니다.

예로 Ups 스크립트 오류가 포함 된 다음과 같은 진화를 만들어 봅시다.

# User에 컬럼을 추가
  
# ---! Ups 
ALTER TABLE userxxx ADD Company varchar ( 255 );
 
# ---! Downs 
ALTER TABLE User DROP Company ;

이 진화의 적용은 실패하고 Play는 데이터베이스 스키마를 불일치 상태로 표시합니다.

계속하려면 먼저이 불일치를 해결해야합니다. 이번에는 올바른 SQL 명령을 실행합니다.

ALTER TABLE User ADD Company varchar ( 255 );

다음 버튼을 클릭하여 불일치를 수동으로 해결 한 것을 Play 알려줍니다.

그러나 진화 스크립트는 아직 잘못된 상태이므로 수정하고 싶은 곳입니다. 그래서 다음과 같이 3.sql 을 편집합니다.

# User에 컬럼을 추가
  
# ---! Ups 
ALTER TABLE User ADD Company varchar ( 255 );
 
# ---! Downs 
ALTER TABLE User DROP Company ;

Play는이 새로운 진화를 감지하여 이전 버전 3으로 대체 해당 스크립트를 실행합니다. 이제 모든 오류가 수정되어 본래의 작업으로 돌아갈 수 있습니다.

그러나 개발 모드에서는 단지 개발을위한 데이터베이스를 삭제하고 모든 진화를 처음부터 다시 적용하는 것이 쉬운 일도 많이 있습니다.
